# sbt 2.0 migration

## Context

sbt 2.0.0 (Scala 3-based metabuild, Bazel-compatible cache) is released. The
previous attempt lives on the stale `update/sbt-2.0.0` branch (forked at
`37f27bb1`), which was blocked because `sbt-scalajs` had no sbt 2.x
cross-build yet. That branch has since diverged badly from `main` (it predates
the websocket / agent / background-task / native-http work), so it is **not**
rebased â€” instead this migration is applied fresh on top of current `main` as
`feature/sbt-2.0-migration`.

The blocker is gone: `sbt-scalajs` (and the rest of the ecosystem, 60+ plugins)
now cross-build to sbt 2.x. In sbt 2.x, cross-building a plugin against Scala
3.x + 2.12.x automatically yields sbt 1.x + sbt 2.x artifacts.

## The real blocker (premise correction)

The Scala.js **core** plugin (`sbt-scalajs` 1.22.0) and Scala Native core
plugin ARE published for sbt 2.x. But this build does not use them directly â€”
it builds on the **portable-scala crossproject family**
(`crossProject(JVMPlatform, JSPlatform, NativePlatform)`, `%%%`,
`CrossType.Pure`). That family â€” `sbt-crossproject`,
`sbt-scalajs-crossproject`, `sbt-scala-native-crossproject`,
`sbt-platform-deps` â€” is **not yet published for sbt 2.x**.

Status upstream: portable-scala/sbt-crossproject PR
[#178 "sbt 2"](https://github.com/portable-scala/sbt-crossproject/pull/178)
(branch `xuwei-k:sbt-2`) â€” CI green, MERGEABLE, but `mergeStateStatus: BLOCKED`
(awaiting maintainer). Open since 2026-04-13.

Second, independent blocker: `scalajs-env-jsdom-nodejs` (used by `domTest`'s
JSDOM env) has no Scala 3 artifact. The sbt 2.x metabuild runs on Scala 3, so
even the sbt-1.x-era `_2.13` jar can't load.

`sbt-projectmatrix` (the only crossproject alternative) is also not yet on
sbt 2.x, so there is no drop-in substitute.

## Unblock steps (when upstream publishes)

1. Bump `sbt-scalajs-crossproject` + `sbt-scala-native-crossproject` to their
   first sbt-2.x release in `project/plugin.sbt`.
2. Bump `scalajs-env-jsdom-nodejs` to a Scala-3 release.
3. `./sbt reload compile test scalafmtAll`, then open the PR.

A local workaround (build all four portable-scala plugins + jsdom from the
unmerged branches via `publishLocal`) would prove the project's own
`build.sbt` is sbt-2-clean, but yields a non-reproducible build that cannot
pass CI â€” not pursued.
