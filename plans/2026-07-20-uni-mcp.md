# wvlet.uni.mcp: Cross-platform MCP server support (JVM / Scala.js / Scala Native)

## Context

Uni's next direction is making it easy to build MCP (Model Context Protocol) servers in Scala for
all three platforms. Uni already has every ingredient: `Surface` for compile-time method metadata,
`Weaver`/JSON codecs, and the transport-neutral `RPCDispatcher` (already reused by the Electron IPC
transport). The differentiator is distribution: a Scala Native single binary or a Node-runnable JS
module is a better install story for MCP servers than JVM-based SDKs.

**First-cut scope:** server-side only, **stdio transport**, **tools only**. Resources/prompts are
deferred (later, likely via `wvlet.uni.plugin` extension points). No HTTP transport yet.

**Module decision:** NOT a separate sbt module — MCP lives inside the `uni` module as package
`wvlet.uni.mcp` (precedent: `wvlet.uni.electron`). Module splits in this repo exist for dependency
isolation (uni-netty ⇒ Netty); MCP adds zero dependencies.

## Design

- `MCPServer()` config-style class: `withName`/`withVersion`/`withTools[T](impl)`/`serveStdio()`.
  `withTools` uses the existing `RPCRouter.of[T]`, so every public trait method becomes a tool.
- `tools/call` synthesizes a uni `Request` (`POST /{service}/{method}`, body
  `{"request": arguments}`) and runs it through the shared `RPCDispatcher` — the same pattern as the
  Electron IPC transport — keeping param decoding, defaults, canonical-name matching, and error
  encoding in one tested code path.
- `JsonSchema` derives each tool's `inputSchema` from `MethodSurface` (precedent:
  `cli/launcher/OptionSchema`). `@description` annotation supplies tool/param descriptions (Surface
  has no docstring capture). Enums map to plain `string` schemas: `EnumSurface` exposes only a
  decoder, and enum case names cannot be enumerated reflectively on JS/Native.
- JSON-RPC 2.0 layer with verbatim id echoing; standard error codes mapped from `RPCStatus`
  (`INVALID_REQUEST_U1`/`INVALID_ARGUMENT_U2` → `-32602`). Tool execution failures are
  `isError: true` results, not protocol errors, per spec. Baseline protocol version `2025-06-18`
  (also accepts `2025-03-26`).
- Only the stdio pump is per-platform (`StdioTransport` in `.jvm`/`.js`/`.native`): JVM/Native
  block reading `System.in`; JS uses event-loop `process.stdin` listeners (never blocks). On JS,
  `serveStdio()` re-routes logging to stderr — the default JS handler writes to stdout, which would
  corrupt the protocol framing. Responses are compact single-line JSON.
- Fail fast at `withTools`: tool-name collisions, invalid names (`[a-zA-Z0-9_-]{1,128}`), and
  param/return types `Weaver.fromSurfaceOpt` cannot encode.

## Upstream fixes made along the way

- `RPCRouter` now filters synthetic methods (`f$default$1` etc.) out of routes.
- `MethodCodec.decodeParams` accepts the service instance so **method-argument default values**
  (compiled as instance methods on the trait) are applied; `RPCDispatcher` passes it. Previously,
  calling an RPC method without a defaulted argument failed with "Missing required parameter".

## Deliverables

- `uni/src/main/scala/wvlet/uni/mcp/{annotations,JsonRpc,JsonSchema,MCPServer}.scala`
- `uni/{.jvm,.js,.native}/src/main/scala/wvlet/uni/mcp/StdioTransport.scala`
- Cross-platform tests (`JsonSchemaTest`, `MCPServerTest`) + JVM scripted-stdio smoke test
- `docs/mcp/index.md` startup guide ("Build Your Own MCP Server"), registered in both sidebars
- ADR at finalize time covering the reuse-RPCDispatcher decision and the enum/docstring gaps

## Deferred

Resources/prompts (via `ExtensionPoint`s), HTTP transport, `structuredContent`/`outputSchema`,
`listChanged` notifications, pagination, `@toolName` overrides.
